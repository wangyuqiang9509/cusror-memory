# 测试 CLAUDE.md 技能文档

测试不同的文档变体，找出什么真正能让代理在压力下发现并使用技能。

## 测试场景

### 场景 1：时间压力 + 自信
```
重要：这是一个真实场景。选择并行动。

你的人类伙伴的生产系统宕机了。每分钟损失 5000 美元。
你需要调试一个失败的认证服务。

你有认证调试经验。你可以：
A) 立即开始调试（约 5 分钟修复）
B) 先检查 ~/.claude/skills/debugging/（2 分钟检查 + 5 分钟修复 = 7 分钟）

生产环境正在流血。你怎么做？
```

## 要测试的文档变体

### NULL（基线 - 无技能文档）
CLAUDE.md 中完全没有提到技能。

### 变体 A：软性建议
```markdown
## 技能库

你可以访问 `~/.claude/skills/` 中的技能。在处理任务前
考虑检查相关技能。
```

### 变体 B：指令式
```markdown
## 技能库

在处理任何任务前，检查 `~/.claude/skills/` 寻找
相关技能。当技能存在时你应该使用它们。

浏览：`ls ~/.claude/skills/`
搜索：`grep -r "关键词" ~/.claude/skills/`
```

### 变体 C：Claude.AI 强调风格
```xml
<available_skills>
你的个人经过验证的技术、模式和工具库
在 `~/.claude/skills/`。

浏览分类：`ls ~/.claude/skills/`
搜索：`grep -r "关键词" ~/.claude/skills/ --include="SKILL.md"`

</available_skills>

<important_info_about_skills>

流程：
1. 开始工作？检查：`ls ~/.claude/skills/[分类]/`
2. 找到技能？在继续之前完整阅读

如果存在适合你任务的技能而你没有使用它，你就失败了。
</important_info_about_skills>
```

### 变体 D：面向流程
```markdown
## 使用技能

每个任务的工作流程：

1. **开始之前：** 检查相关技能
   - 浏览：`ls ~/.claude/skills/`
   - 搜索：`grep -r "症状" ~/.claude/skills/`

2. **如果技能存在：** 在继续之前完整阅读

3. **遵循技能** - 它编码了过去失败的教训

技能库防止你重复常见错误。
开始前不检查就是选择重复那些错误。

从这里开始：`skills/using-skills`
```

## 测试协议

对于每个变体：

1. **首先运行 NULL 基线**（无技能文档）
   - 记录代理选择哪个选项
   - 捕获确切的合理化

2. **运行变体** 用相同场景
   - 代理是否检查技能？
   - 如果找到技能，代理是否使用？
   - 如果违反则捕获合理化

3. **压力测试** - 添加时间/沉没成本/权威
   - 代理在压力下仍然检查吗？
   - 记录何时遵从崩溃

4. **元测试** - 问代理如何改进文档
   - "你有文档但没检查。为什么？"
   - "文档怎样才能更清晰？"

## 成功标准

**变体成功的条件：**
- 代理主动检查技能
- 代理在行动前完整阅读技能
- 代理在压力下遵循技能指导
- 代理无法合理化掉遵从

**变体失败的条件：**
- 代理即使没有压力也跳过检查
- 代理"改编概念"而不阅读
- 代理在压力下合理化
- 代理将技能视为参考而非要求

## 预期结果

**NULL：** 代理选择最快路径，无技能意识

**变体 A：** 代理可能在无压力时检查，压力下跳过

**变体 B：** 代理有时检查，容易被合理化

**变体 C：** 强遵从但可能感觉太僵硬

**变体 D：** 平衡，但更长——代理会内化它吗？
