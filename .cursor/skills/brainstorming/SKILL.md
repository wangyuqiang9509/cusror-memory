# 将创意转化为设计方案

## 概述

通过自然的协作对话，帮助将创意转化为完整的设计和规格说明。

首先了解当前项目的背景，然后逐一提问以完善创意。一旦理解了要构建的内容，就将设计分成小部分（200-300字）呈现，并在每个部分之后确认是否看起来正确。

## 流程

**理解创意：**
- 首先查看当前项目状态（文件、文档、最近的提交）
- 逐一提出问题以完善创意
- 尽可能使用选择题，但开放式问题也可以
- 每次消息只问一个问题——如果一个主题需要更多探索，将其分解为多个问题
- 专注于理解：目的、约束条件、成功标准

**探索方案：**
- 提出2-3种不同的方法及其权衡
- 以对话方式呈现选项，包含你的建议和理由
- 首先提出你推荐的选项并解释原因

**呈现设计：**
- 一旦你理解了要构建的内容，就呈现设计方案
- 将其分解为200-300字的小节
- 在每个部分之后询问是否看起来正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 准备好返回并澄清如果某些内容没有意义

## 设计之后

**文档：**
- 将验证后的设计写入 `docs/plans/YYYY-MM-DD-&lt;主题&gt;-design.md`
- 如果有可用的 writing-clearly-and-concisely 技能，请使用它

**实现（如果继续）：**
- 询问："准备好开始实现了吗？"
- 使用 writing-plans 创建详细的实现计划

## 关键原则

- **一次一个问题**——不要用多个问题压倒对方
- **优先选择题**——比开放式问题更容易回答
- **严格遵循YAGNI**——从所有设计中移除不必要的功能
- **探索替代方案**——在确定之前总是提出2-3种方法
- **增量验证**——分部分呈现设计，逐个验证
- **保持灵活**——当某些内容没有意义时，返回并澄清
